Le fonctionnement du Financial Command Center, expliqué factuellement. La navigation suit le cycle de vie d'une banque : entrée en relation (KYC) → entrée des opérations (Ingestion) → activité de marché → post-marché (Réconciliation) → Comptabilité → analyse (Explorateur) → restitution (Rapports).

Le projet

Qu'est-ce que ce système ?

Un Data Mesh bancaire : 7 « Data Products » autonomes (trading, trésorerie, risque, comptabilité, client/KYC, audit, regulatory), chacun sous contrat versionné, gouvernés par 10 règles appliquées par le code (G1–G10, voir docs/governance.md). Tout ce que tu vois sur ce site est calculé depuis ces produits — jamais saisi.

Les données sont-elles réelles ?

Non : la provenance actuelle est simulée (générateur déterministe, réaliste pour une salle de marchés). La règle G8 rend cette frontière structurelle : la provenance est portée par chaque lot, propagée à tout dérivé, et aucune publication réglementaire réelle n'est possible depuis du simulé (seuls des DRYRUN sortent). Brancher des données réelles = une source origin=production, rien d'autre ne change.

Entrées

Comment entrer mes propres données ?

Page Ingestion : colle un CSV ou choisis un fichier. Le DataTransformer mappe vers l'ontologie, valide contre le contrat, motive chaque rejet, et scelle le SHA-256 de l'entrée dans le journal d'audit. Pour garantir l'exhaustivité, fournis les totaux de contrôle (control_totals : nombre de lignes + sommes par devise) — un écart fait refuser le fichier en totalité.

Pourquoi une notation KYC est-elle low / medium / high ?

La notation est dérivée de règles déclaratives, jamais arbitraire : high = PEP déclaré ou juridiction à vigilance renforcée ; medium = établissement non bancaire (surveillance prudentielle moindre) ; low = banque régulée en juridiction standard. La justification exacte de chaque dossier est affichée sous son nom sur la page KYC/AML. PEP = personne politiquement exposée (obligations LBC-FT, vigilance renforcée).

Contrôles & confiance

Comment être sûr qu'aucune donnée ne se perd ?

Cinq maillons contrôlés par le code (docs/data-quality-trust.md) : totaux de contrôle à l'entrée ; partie double comme détecteur (tout flux inexpliqué atterrit au compte d'attente 9990, affiché en rouge) ; six assertions d'audit par produit et par jour ; contrôles de restitution bloquants sur chaque rapport ; preuve re-vérifiable (journal chaîné SHA-256).

L'IA peut-elle modifier mes données ?

Non. Les IA du système (réconciliation, pré-diagnostic AML) proposent des scores explicables avec leur lineage (G6) ; seule une décision humaine (Accepter/Rejeter, Escalader/Classer) change un état — et elle est journalisée puis apprise par la boucle de feedback.

Pourquoi un export m'est-il refusé ?

Trois causes, toutes tracées : habilitation insuffisante (G9 — ton rôle ne couvre pas la classification de la donnée) ; assertion d'audit manquante ou non certifiée (G10) ; contrôle de restitution en échec (totaux qui ne bouclent pas). Le refus exact s'affiche et se journalise.

Sorties

Comment générer un rapport certifié ?

Page Rapports : choisis un template (EMIR, MiFID II, FINREP F 01.01 FR/EN, bilan, trésorerie…), un format (CSV/XLSX/PDF) et un rôle. Chaque livrable embarque son Annexe de Preuve : norme source, horodatage UTC, demandeur, SHA-256, assertions, contrôles. Le bilan et le PnL s'exportent aussi depuis la page Comptabilité.

Comment vérifier un chiffre six mois plus tard ?

Du document au hash (Annexe de Preuve), du hash à l'entrée de journal (verify_chain), de l'entrée à l'assertion, de l'assertion aux données sources — chaque maillon est re-calculable sans intervention humaine.

Comment lancer l'application ?

En local : git clone, puis python3 -m app → localhost:8787 (ajoute pip3 install duckdb pour l'explorateur SQL). En ligne : le site est republié automatiquement sur GitHub Pages à chaque merge sur main.
